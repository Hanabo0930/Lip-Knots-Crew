# 本番SLO・自動インシデント管理 v3.4

## SLO基準

| 対象 | 初期値 |
|---|---:|
| 総合可用性 | 99.9% |
| 認証成功率 | 99.5% |
| Functions成功率 | 99.5% |
| スプシ書込成功率 | 99.0% |
| 通知成功率 | 98.0% |
| p95応答時間 | 2,000ms以下 |
| 最古queue滞留 | 15分以下 |
| 観測鮮度 | 15分以内、30分超でSEV2 |

可用性の許容失敗率を30日エラーバジェットとし、1h・6h・24h・30dの時間窓を評価します。基準変更はインシデント中にできません。

## 自動判定

- `SEV1`: 重大機能停止、データ差異、1時間14.4倍以上の高速バーン
- `SEV2`: 6時間6倍、24時間3倍、予算100%以上、重大遅延・queue・監視断
- `SEV3`: 予算80%以上、個別成功率低下、軽度遅延・queue・監視probe

異常を検出すると企業ごとに一つの進行中インシデントへ自動起票・統合します。SEV上昇、失敗signal構成の変化、復旧確認だけをPushし、同じ異常値の増減では通知を連打しません。

## 状態遷移

`open` → `acknowledged` → `monitoring_recovery` → `recovery_pending` → `resolved`

- 担当者と10文字以上の初動メモでacknowledge
- 新しい正常観測を3回連続で`recovery_pending`
- 5分定期監視の再評価だけでは正常回数を増やさない
- 新しい異常で正常回数を0へ戻し、必要ならSEVを上げる
- 原因・復旧・再発防止を各20文字以上、証跡2件以上で解決

## データ保持

生観測、時間bucket、最新snapshot、incident、event、auditLogsを保持します。時間bucketは最大721時間程度の読取で30日を評価し、生観測を毎回全走査しません。

## 実運用

Cloud Monitoring等から5〜15分単位で集計した件数を管理画面またはCallableへ入力します。本番有効化直後に最初の観測を登録し、監視断による自動起票を防ぎます。
